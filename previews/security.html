// SECURITY

How we secure data, secrets, and the appeal trail.

// data_classification

WALLET_ADDRESSES

Public on-chain data. We process but never re-publish individual addresses outside aggregate stats.

EMAIL + KEY_HASH

SHA256-hashed in DB. Plaintext keys shown ONCE at creation.

EVIDENCE + CLUSTERS

Visible only to the customer who created the analysis. Aggregates may appear in public retros.

APPEAL_SUBMISSIONS

Visible to analysis owner + reviewer. Submitter contact email never published.

WEBHOOKS

Payloads signed with HMAC-SHA256 against the customer's webhook_secret. Verify before processing.

AUDIT_LOG

Append-only. No admin can edit prior entries. Schema captures actor, prior/new score, evidence snapshot.

// api_key_handling

  • › Format: sk_live_<32 base64url chars>
  • › SHA256-hashed in DB. Plaintext shown once at creation/rotation.
  • › Constant-time comparison on every request (timingSafeEqual).
  • › Rotate at /dashboard/api-keys — old key revoked immediately.

// webhook_signature

On analysis.completed we POST to your URL with header x-sybilshield-signature: sha256=<hex>. Recompute the HMAC over the raw request body and compare with constant-time equality. Reject mismatches.

// audit_log_format

Every flagged event (score ≥ 40), every appeal, every review, every reversal is written. Rows are append-only. Schema captures:

  • › actor — system:model:vX | customer:<uuid> | public:anonymous
  • › timestamp · prior_score · new_score
  • › evidence_snapshot — JSON, frozen at event time
  • › reason — free text

// vulnerability_disclosure

Email security@sybilshield.org with a description and reproduction.

  • › Acknowledge within 48 hours
  • › Patch critical issues within 7 days
  • › Public-credit you in the disclosure if you wish
  • › No legal action against good-faith research
